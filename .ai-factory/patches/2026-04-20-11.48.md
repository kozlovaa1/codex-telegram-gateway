# Handle preflight failures in session restart/reset commands

**Date:** 2026-04-20 11:48
**Files:** src/codex_telegram_gateway/app.py, tests/test_app_session_controls.py
**Severity:** high

## Problem

Telegram commands that reset or restart a workspace session could fail silently for
unwritable workspaces. Manual QA reproduced this with `/session restart` and
`/newsession` on `/srv/infra`: the bot sent no reply, session state did not change,
and logs showed `Task exception was never retrieved`.

There was also a user-facing naming leak on the happy path: successful restart/reset
messages exposed internal workspace ids such as
`session:334845402:434774:project:codex-telegram-gateway` instead of the display name.

## Root Cause

The Telegram command path called `SessionManager.restart_workspace(...)`, which can
raise `WorkspacePreflightError`, but `_session_command(..., ["restart"])` and
`_reset_session(...)` did not catch that exception. The prompt execution path already
translated preflight failures into user-safe Telegram replies, but the command path
did not, so the exception escaped the task and produced no bot response.

The naming leak happened because these same command handlers interpolated the raw
workspace name directly into reply text instead of normalizing through
`display_workspace_name(...)`.

## Solution

Added targeted exception handling in `app.py` for session restart/reset commands:

- catch `WorkspacePreflightError`
- log the failure with `[FIX]`-prefixed structured warning entries
- send `exc.result.user_message` back to the Telegram user
- return early without leaving the task unhandled

Also changed the success messages for restart/reset to use
`display_workspace_name(name)` so internal session workspace ids are not exposed in
bot replies.

Added regression tests covering:

- user-facing preflight error on `/session restart`
- user-facing preflight error on `/newsession` / reset path
- hidden internal workspace name on restart success
- hidden internal workspace name on reset success

## Prevention

- Any Telegram command path that calls infrastructure code capable of raising
  boundary exceptions must convert them into user-safe replies at the app layer.
- Review user-facing responses for internal ids before shipping topic/thread-scoped
  session features.
- Keep manual QA for both happy-path writable workspaces and failing preflight
  workspaces; unit tests alone would not have surfaced the silent bot behavior as
  clearly.
