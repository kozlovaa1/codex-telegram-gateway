## Tasks

### Phase 1: Foundations
- [x] Task 1: Add explicit UX configuration and policy primitives in `src/codex_telegram_gateway/config.py`, `src/codex_telegram_gateway/models.py`, `config.example.toml`, and `config.toml` so the gateway can decide `reaction + typing + progress + stream` for private chats and `reaction + typing + final-only` for groups/topics without ad hoc checks in `app.py`. This task must extend `AppConfig` via a dedicated typed UX settings block plus any nested settings types needed for policy resolution, validate incompatible UX flag combinations during config load, keep omitted options fail-safe, and update the existing `AppConfig(...)` test fixtures plus `load_config()` validation coverage in `tests/test_config.py`, `tests/test_main.py`, `tests/test_execution_policy.py`, `tests/test_session_manager.py`, `tests/test_app_default_workspace.py`, and `tests/test_app_session_controls.py`. Logging requirements: log config parsing/validation failures, log effective UX feature flags per request at DEBUG, and keep defaults safe when a UX option is omitted.
- [x] Task 2: Extend `src/codex_telegram_gateway/telegram_api.py` with transport helpers for message reactions, chat actions, and the minimal send/edit primitives the UX layer will need, plus Telegram feature/capability normalization and typed error classification. Keep this task transport-scoped: it should not own runtime lifecycle or fallback orchestration logic that belongs in the UX layer. Explicitly include Telegram HTTP/body error parsing, retry-after extraction, capability probing/downgrade rules, and safe wrappers for `sendChatAction`, reactions, and send/edit fallbacks. Logging requirements: log `reaction_sent`, `reaction_failed`, `typing_started`, transport capability downgrades, rate-limit/timeouts, and whether a failure is treated as ignorable fallback vs final-delivery risk.
- [x] Task 3: Add an early prompt-path regression harness around `GatewayApp._handle_prompt()` before the larger UX refactor so the current queued/edit/final flow is covered while behavior is being reshaped. This task should add focused tests for the initial acknowledgement path, streaming edit throttling, preflight/policy/internal-error branches, edit failure fallback expectations, and guaranteed final delivery boundaries using small fakes rather than the final UX implementation. Files should include a new or expanded app-level test module such as `tests/test_app_prompt_flow.py` plus any minimal test helpers needed. Logging requirements: tests should assert observable fallback behavior where practical so later refactors cannot silently regress the Telegram reply path.

### Phase 2: UX Runtime Layer
- [x] Task 4: Introduce a dedicated response UX module such as `src/codex_telegram_gateway/response_ux.py` to own `ResponseUxPolicy`, `TelegramFeatureSupport`, per-response lifecycle state, and a single orchestration surface that `GatewayApp._handle_prompt()` can call. This task must define a concrete event contract between `src/codex_telegram_gateway/codex_adapter.py`, `src/codex_telegram_gateway/session_manager.py`, and the UX layer so private progress/stream rendering does not depend on raw Telegram edit logic inside `app.py`. It must also introduce a typed request/response context model in `src/codex_telegram_gateway/models.py` for inbound message identity, reply target, chat capability, and policy selection so duplicate responder protection is keyed to a concrete request identity (`chat_id`, `thread_id`, `message_id`, or an equivalent request id) instead of loose argument passing. The UX layer must own a request-scoped lifecycle registry and terminal cleanup on success, failure, cancellation, `/stop`, `/newsession`, and `/session restart`. Files should include the new module plus focused wiring changes in `src/codex_telegram_gateway/app.py`. Logging requirements: emit `inbound_message_accepted`, resolved chat capability/policy, lifecycle start/stop events, and guard against duplicate responders for the same inbound message.
- [x] Task 5: Wire the new UX layer into bootstrap and runtime construction by updating `src/codex_telegram_gateway/__main__.py`, any `GatewayApp` constructor surfaces, and `tests/test_main.py` so response UX dependencies are created once and injected explicitly instead of being assembled ad hoc inside `app.py`. Depends on Task 1 and Task 4. Keep this task focused on composition/wiring rather than delivery behavior. Logging requirements: log UX bootstrap success/failure and the enabled policy surface during startup without duplicating per-request runtime logs.
- [x] Task 6: Implement `TypingHeartbeat` and reaction acknowledgement in the UX layer so accepted prompts stop using the text message `[workspace] queued` and instead use early reaction plus periodic `sendChatAction(typing)` with clean shutdown on success, error, cancellation, `/stop`, and session reset. Depends on Task 1, Task 2, and Task 4. Files should include the new UX module, `app.py`, and any minimal `session_manager.py` hooks needed for cancellation-safe cleanup. Logging requirements: log heartbeat start/stop, skipped sends due to throttle, unsupported-context fallback, and cleanup on cancellation without leaking background tasks.

### Phase 3: Private vs Group Delivery
- [ ] Task 7: Add normalized run-event shaping and cleanup coverage at the adapter/session boundary by extending `src/codex_telegram_gateway/codex_adapter.py`, `src/codex_telegram_gateway/session_manager.py`, `tests/test_codex_adapter.py`, and `tests/test_session_manager.py` so the UX layer consumes stable event types and cleanup hooks rather than raw transport noise. Depends on Task 4. This task exists because the response UX should not be forced to parse arbitrary JSON event payloads directly from `CodexAdapter.run()`, and regressions here could break progress or streaming even when app-level tests still pass.
- [ ] Task 8: Build a throttled `ProgressAggregator` / `ProgressReporter` for private chats only, consuming Codex event stream data from `src/codex_telegram_gateway/codex_adapter.py` / `SessionManager.execute()` and translating internal events into short user-facing stages instead of raw tool spam. Depends on Task 4 and Task 7. Files should include the UX module plus any small event-shaping helpers in `codex_adapter.py` or `models.py`, with the event-shaping boundary explicit enough that the UX layer consumes normalized run events instead of parsing arbitrary raw adapter payloads inside `app.py`. Logging requirements: log `progress_update_sent`, `progress_update_skipped`, aggregation decisions, dropped noisy events, and fallback disablement if progress rendering starts failing.
- [ ] Task 9: Build a private-chat `StreamingResponder` abstraction with throttled chunk accumulation, Telegram length protection, editable/draft-first behavior when supported, and fallback to a simpler editable/private message flow when streaming support or edits fail. The responder must be private-chat only by policy, handle missing/non-editable placeholder messages, create a fresh message when edits fail, and preserve safe chunk splitting for long answers. Depends on Task 2, Task 4, and Task 7. Files should include the UX module, `telegram_api.py`, and `app.py`. Logging requirements: log `stream_started`, `stream_chunk_sent`, `stream_failed`, `stream_fallback_used`, edit conflicts, message length splits, and Telegram retry-after handling.
- [ ] Task 10: Refactor finalization so private chats finalize the active progress/stream state cleanly while groups and topics always send one final answer only, with UX helper failures never preventing final delivery. Depends on Task 1, Task 6, Task 8, and Task 9. Files should include `src/codex_telegram_gateway/app.py`, the UX module, and possibly `session_manager.py` if explicit stop/reset hooks are required. Logging requirements: log final-delivery path selection, `final_response_sent`, `final_response_failed`, final fallback chain selection, and cleanup of temporary response state after success, failure, or cancellation.

This task exists because `src/codex_telegram_gateway/telegram_api.py` is a boundary adapter and should be validated independently from app orchestration before UX rollout is treated as complete.
